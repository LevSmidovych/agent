# Scrum: основи

Стисло по ролях, артефактах і подіях. Офіційний Scrum Guide 2020 року.

## Три ролі

**Product Owner (PO)**
- Відповідає за цінність продукту
- Веде Product Backlog, пріоритезує
- Єдина особа яка приймає рішення що робити
- Говорить з замовниками і стейкхолдерами

**Scrum Master (SM)**
- Допомагає команді слідувати Scrum
- Прибирає блокери
- Коучить команду і PO
- НЕ менеджер і НЕ керівник команди

**Developers**
- Команда яка робить роботу
- Самоорганізується
- Крос-функціональна: всі скіли потрібні для поставки інкременту є всередині
- Рекомендований розмір: 3-9 осіб

## Артефакти

**Product Backlog**
- Список всього що треба зробити
- Впорядкований за пріоритетом
- Елементи зверху деталізовані, знизу можуть бути абстрактні
- Власник: PO

**Sprint Backlog**
- Що команда взяла на спринт
- Складається з частини Product Backlog + план реалізації
- Власник: Developers

**Increment**
- Готовий робочий продукт в кінці спринта
- Має відповідати Definition of Done

## Події

**Sprint** - контейнер для всього іншого. Фіксована тривалість, 1-4 тижні. Найчастіше 2 тижні.

**Sprint Planning** - початок спринта. Максимум 8 годин для місячного спринта, пропорційно менше для коротших. Команда вирішує що зробить і як.

**Daily Scrum** - щоденний стендап, 15 хвилин. Для Developers, не статус-мітинг для менеджера. Три питання класичні (що зробив, що зроблю, блокери) вже не обовʼязкові в новому гайді - формат команда обирає сама.

**Sprint Review** - в кінці спринта. Демо, зворотний звʼязок від стейкхолдерів, оновлення беклогу. Максимум 4 години для місячного спринта.

**Sprint Retrospective** - після ревʼю. Команда обговорює як покращити процес. Максимум 3 години.

## Definition of Done (DoD)

Список критеріїв коли задача вважається готовою. Приклад для розробки:
- Код написаний і ревʼю пройдено
- Юніт-тести написані, покриття не знижено
- Інтеграційні тести проходять
- Документація оновлена
- Деплой на staging зроблено
- QA прийняв

DoD формулює команда, не PO.

## Типові помилки

- Scrum Master як проджект-менеджер: він не розподіляє задачі
- Зміна scope посеред спринта: PO не має права це робити, тільки скасувати спринт повністю
- Стендап для менеджера: мітинг для команди, а не звіт нагору
- Ретро без дій: обговорили проблеми, нічого не змінили - марно
